# 6. 服务器运维手册

本文描述 **fancy-print** 云端（`cloud/`）在 Linux 服务器上的 **部署、日常运维、排障与回滚** 约定，与 [`4. 服务器端设计.md`](4.%20服务器端设计.md) **§7 部署与环境** 及 [`../cloud/README.md`](../cloud/README.md) 互补。

---

## 1. 架构与入口

| 组件 | 默认端口 | 说明 |
|------|-----------|------|
| **gateway** | 3000 | 对外统一入口：`/v1/parent/*` → parent-bff，其余 `/v1/*` → device-api |
| **device-api** | 3001 | 设备 HTTPS API、Job 流水线、可选 MQTT |
| **parent-bff** | 3002 | 家长 BFF（开发桩登录 / OIDC 扩展见 `cloud/README.md`） |
| **Redis** | 6379（仅容器内） | 多副本 Job 状态、幂等、BullMQ 队列（`PIPELINE_QUEUE_BACKEND=bullmq`） |

**健康检查**：各服务均为 **`GET /health`**（200）。Prometheus 文本指标：**`GET /metrics`**（gateway / device-api / parent-bff）。

**一键全栈**对外只须暴露 **gateway:3000**；3001/3002 仅用于联调或内网管理，生产建议在 Compose 中去掉 `ports` 映射。

---

## 2. 前置条件

- **操作系统**：推荐 Ubuntu 22.04 LTS / Debian 12+（x86_64 或 arm64）。
- **Docker**：24+，且已安装 **Docker Compose v2**（`docker compose version`）。
- **Shell**：`bash` 5+；脚本在 **WSL2、macOS、Linux** 下可用；纯 Windows CMD 请用 WSL 执行同一脚本。
- **网络**：构建阶段需能访问 **npm registry**（`npm ci`）；运行期按业务访问公网 ASR/生图等。
- **资源（经验值）**：最小演示约 **2 vCPU / 4 GiB RAM**；含 BullMQ worker 与并发生图时请按并发量上调。

## 3. 一键部署（推荐）

### 3.1 首次部署（全栈）

在 **克隆后的仓库根目录** 执行：

```bash
chmod +x cloud/deploy/scripts/*.sh
cp cloud/deploy/env/stack.example.env cloud/deploy/env/stack.env
# 编辑 stack.env：至少修改所有 JWT / 设备密钥为强随机串
./cloud/deploy/scripts/deploy.sh
```

脚本会：

1. 使用 `cloud/deploy/compose/docker-compose.stack.yml` **构建镜像**并 **`up -d --build`**；
2. 若不存在 `cloud/deploy/env/stack.env`，会从 **`stack.example.env`** 复制一份（**务必改密钥后再用于生产**）。

部署成功后：

- 网关：`http://<主机>:3000/health`
- 直连 device-api：`http://<主机>:3001/health`
- 直连 parent-bff：`http://<主机>:3002/health`

### 3.2 仅设备通道演示栈（无 gateway / parent-bff）

适合只验证 Job + Redis + BullMQ worker：

```bash
./cloud/deploy/scripts/deploy.sh --device-only
```

对应 Compose：`cloud/deploy/compose/docker-compose.device-api.yml`。

### 3.3 带代码更新的重新部署

在仓库内更新代码后：

```bash
./cloud/deploy/scripts/deploy.sh --pull          # 先 git pull --ff-only 再全栈部署
# 或
./cloud/deploy/scripts/upgrade.sh                # 当前等价于 deploy.sh
```

## 4. 运维脚本一览

| 脚本 | 作用 |
|------|------|
| [`../cloud/deploy/scripts/deploy.sh`](../cloud/deploy/scripts/deploy.sh) | 一键构建并启动（全栈或 `--device-only`；可选 `--pull`） |
| [`../cloud/deploy/scripts/stop.sh`](../cloud/deploy/scripts/stop.sh) | 停止栈；`--volumes` 删除 Redis 数据卷；`--device-only` 对应设备演示 compose |
| [`../cloud/deploy/scripts/logs.sh`](../cloud/deploy/scripts/logs.sh) | `docker compose logs -f`；可附加服务名，如 `logs.sh gateway` |
| [`../cloud/deploy/scripts/healthcheck.sh`](../cloud/deploy/scripts/healthcheck.sh) | `curl` 检查 `/health`；可通过环境变量改 URL（见脚本内注释） |
| [`../cloud/deploy/scripts/upgrade.sh`](../cloud/deploy/scripts/upgrade.sh) | 转调 `deploy.sh`（便于运维别名） |
| [`../cloud/deploy/scripts/lib/common.sh`](../cloud/deploy/scripts/lib/common.sh) | 路径与 `docker compose` 封装（被其他脚本 source） |

**项目名**：默认 Compose 项目名为 **`fancy-print`**，可通过环境变量 **`FP_PROJECT`** 覆盖。

**自定义路径**：

- `FP_ENV_STACK`：全栈 env 文件（默认 `cloud/deploy/env/stack.env`）
- `FP_COMPOSE_STACK` / `FP_COMPOSE_DEVICE_API`：compose 文件路径

## 5. 环境变量与密钥

### 5.1 全栈 `stack.env`（模板：`cloud/deploy/env/stack.example.env`）

| 变量 | 用途 |
|------|------|
| `DEVICE_JWT_ACCESS_SECRET` / `DEVICE_JWT_REFRESH_SECRET` | device-api 设备 JWT |
| `PARENT_JWT_ACCESS_SECRET` / `PARENT_JWT_REFRESH_SECRET` | parent-bff JWT |
| `PARENT_DEV_PASSWORD` | 开发桩家长登录密码（默认 `dev`） |
| `PIPELINE_QUEUE_CONCURRENCY` | device-api 内 BullMQ 并发槽 |
| `PIPELINE_WORKER_CONCURRENCY` | 独立 pipeline worker 并发槽 |

全栈 Compose 已为 **device-api** 设置 **`REDIS_URL`**、**`PIPELINE_QUEUE_BACKEND=bullmq`**；**gateway** 使用服务名 **`http://device-api:3001`**、**`http://parent-bff:3002`**。

### 5.2 生产补强（按需）

- **TLS / mTLS**：见 `cloud/docs/ingress-mtls.md` 与 gateway 的 `GATEWAY_TLS_*` 等（进程内 HTTPS 或前置 Nginx/Ingress）。
- **厂商与对象存储**：`IFLYTEK_*`、`WANX_*`、`S3_*`、`MODERATION_*` 等见 [`../cloud/README.md`](../cloud/README.md) 与 OpenAPI。
- **MQTT**：`MQTT_URL`、`MQTT_RECONNECT_MS` 等见 `cloud/README.md`。

**切勿**将 `stack.env` 提交到 Git（仓库已 `.gitignore` **`cloud/deploy/env/stack.env`**）。

## 6. 日常运维

### 6.1 查看状态与日志

```bash
docker compose --project-name fancy-print ps
./cloud/deploy/scripts/logs.sh
./cloud/deploy/scripts/logs.sh gateway device-api
```

### 6.2 停机与清理

```bash
./cloud/deploy/scripts/stop.sh
# 清空 Redis 持久化卷（危险）
./cloud/deploy/scripts/stop.sh --volumes
```

### 6.3 仅重启某一服务（示例）

```bash
docker compose --project-name fancy-print \
  -f cloud/deploy/compose/docker-compose.stack.yml \
  --env-file cloud/deploy/env/stack.env \
  restart gateway
```

### 6.4 备份与 Redis

当前 Compose 为 Redis 挂载 **`redis-data`** 卷（AOF）。备份思路：

1. **维护窗口**内 `stop.sh`（不停 Redis 也可）后 `docker run --rm -v fancy-print_redis-data:/data -v $(pwd):/backup alpine tar czf /backup/redis-data.tgz /data`；或  
2. 使用托管 **Redis**（云厂商）并将 **`REDIS_URL`** 指向托管实例，Compose 内去掉内置 `redis` 服务（需自行改 compose，属进阶定制）。

Job 冷迁移 / 关机导出见 **`JOB_REDIS_IMPORT_FILE`**、**`JOB_FILE_EXPORT_ON_SHUTDOWN`**（`cloud/README.md`）。

## 8. 排障清单

| 现象 | 排查 |
|------|------|
| `deploy.sh` 构建失败 | 宿主机 Docker 磁盘空间、`npm ci` 网络；查看构建日志末段 |
| gateway 502 / 空响应 | `logs.sh gateway`；确认 `DEVICE_API_URL` / `PARENT_BFF_URL` 在容器内可解析（compose 默认已写服务名） |
| device-api 启动即退出 | `logs.sh device-api`；常见为缺 **`REDIS_URL`** 或 JWT 环境变量为空 |
| Job 不推进 | 是否配置 **`POST /v1/jobs/{id}/advance`**；多副本是否 **`REDIS_URL`** + **`PIPELINE_QUEUE_BACKEND=bullmq`** |
| 健康检查失败 | `./cloud/deploy/scripts/healthcheck.sh`；防火墙是否只放行 3000 |
